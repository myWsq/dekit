# Dekit — Agent Usage Guide

Dekit is a design canvas for code agents. You create HTML/CSS designs, dekit renders them, and you take screenshots to see the result.

## Project Structure

All design files live inside a `.dekit/` directory:

```
project/
└── .dekit/
    ├── dekit.yaml          ← config (pages, components, global style)
    ├── global.css           ← shared styles across all pages
    ├── pages/
    │   └── home/
    │       ├── home.html    ← page template (plain HTML, no framework)
    │       └── home.css     ← page-specific styles
    ├── components/
    │   └── ui-card/
    │       ├── ui-card.html ← Web Component template
    │       └── ui-card.css  ← component scoped styles
    └── screenshots/         ← generated by dekit screenshot
```

## Quick Start

```bash
# 1. Initialize (creates .dekit/ in current directory)
dekit init --template landing

# 2. Take a screenshot to see the design
dekit screenshot '$${home}'

# 3. The screenshot is saved to .dekit/screenshots/home.png
# Read the screenshot file to see the result, then iterate
```

## Writing Pages

Pages are plain HTML — no framework, no build step. Write standard HTML tags and CSS.

**Page HTML** (`.dekit/pages/home/home.html`):
```html
<header class="hero">
  <h1>Welcome</h1>
  <p>A short description.</p>
  <a class="cta" href="#">Get Started</a>
</header>

<section class="features">
  <div class="feature">
    <h3>Fast</h3>
    <p>Lightning quick performance.</p>
  </div>
</section>
```

**Page CSS** (`.dekit/pages/home/home.css`):
```css
.hero {
  display: flex;
  flex-direction: column;
  align-items: center;
  padding: 96px 24px;
  gap: 16px;
}

.cta {
  padding: 12px 32px;
  background: var(--color-primary);
  color: #fff;
  border-radius: 8px;
  text-decoration: none;
}
```

Key rules:
- Write the `<body>` content only — no `<html>`, `<head>`, or `<body>` tags (dekit wraps it)
- Use CSS variables from `global.css` (e.g. `var(--color-primary)`)
- Each page has its own HTML + CSS file pair

## Writing Components

Components use Web Components (custom elements with Shadow DOM). The tag name **must contain a hyphen**.

**Component HTML** (`.dekit/components/ui-card/ui-card.html`):
```html
<template>
  <div class="card">
    <div class="card-image">
      <slot name="image"></slot>
    </div>
    <div class="card-body">
      <slot></slot>
    </div>
  </div>
</template>
```

**Component CSS** (`.dekit/components/ui-card/ui-card.css`):
```css
.card {
  border: 1px solid #e5e7eb;
  border-radius: 12px;
  overflow: hidden;
}

.card-body {
  padding: 16px;
}
```

**Using components in pages:**
```html
<ui-card>
  <img slot="image" src="photo.jpg">
  <h3>Title</h3>
  <p>Description</p>
</ui-card>
```

Key rules:
- Wrap content in `<template>` tags
- Use `<slot></slot>` for default content, `<slot name="x"></slot>` for named slots
- CSS is scoped to the component (Shadow DOM) — styles don't leak
- Use `:host` selector to style the component element itself
- Use `::slotted(*)` to style slotted content

## dekit.yaml

Every page and component must be registered:

```yaml
version: 1.0
global-style: "global.css"

components:
  ui-card:
    template: "components/ui-card/ui-card.html"
    style: "components/ui-card/ui-card.css"

pages:
  home:
    template: "pages/home/home.html"
    style: "pages/home/home.css"
  about:
    template: "pages/about/about.html"
    style: "pages/about/about.css"
```

All paths are relative to the `.dekit/` directory.

## CLI Reference

### dekit init [path] [--template name]

Initialize a design project. Creates `.dekit/` directory under the target path (default: current directory).

Templates: `blank`, `landing`, `dashboard`, `mobile`.

```bash
dekit init                           # .dekit/ in current directory
dekit init my-project                # my-project/.dekit/
dekit init --template landing        # With template
dekit init --template                # List available templates
```

### dekit add page \<name\> [--template name]

Add a page. Creates files and updates dekit.yaml.

Templates: `blank`, `hero`, `form`, `grid`.

```bash
dekit add page about
dekit add page pricing --template grid
```

### dekit add component \<name\>

Add a Web Component (name must contain a hyphen). Creates files and updates dekit.yaml.

```bash
dekit add component my-card
```

### dekit ls

List all pages and components in the project.

### dekit serve [-p port] [--no-open]

Start the interactive preview server (default port: 7980). Use this when a human wants to review the design in a browser.

### dekit screenshot \<ref\>

Take a screenshot. The output file path is printed to stdout — read that file to see the result.

```bash
dekit screenshot '$${home}'                    # Full page → .dekit/screenshots/home.png
dekit screenshot '$${home@.hero}'              # Element only
dekit screenshot '$${home}' --device iphone-14 # Device size
dekit screenshot '$${home}' --full-page        # Include scroll area
dekit screenshot '$${home}' -o output.png      # Custom output path
dekit screenshot --all                         # All pages
```

### dekit resolve \<ref\>

Resolve a ref to the source file path and line range. Use this when a human gives you a ref and you need to find the code to edit.

```bash
dekit resolve '$${home}'
# → .dekit/pages/home/home.html

dekit resolve '$${home@.hero}'
# → .dekit/pages/home/home.html:1-10
```

### dekit usage

Print this guide.

## Ref Protocol

Refs identify pages and elements. Format: `$${page-key}` or `$${page-key@css-selector}`.

When you see a `$${...}` pattern in user messages:
- `dekit resolve <ref>` → find the source file and line range to edit
- `dekit screenshot <ref>` → take a screenshot of that page or element

You don't need to parse refs yourself — just pass them to the CLI.

## Design Workflow

### Self-directed design loop

```
1. dekit init --template landing          # Start project
2. Edit .dekit/pages/home/home.html       # Write HTML
3. Edit .dekit/pages/home/home.css        # Write CSS
4. dekit screenshot '$${home}'            # Take screenshot
5. Read the screenshot file               # Analyze the result
6. Edit HTML/CSS to fix issues            # Iterate
7. Repeat 4-6 until satisfied             # Keep improving
8. dekit serve                            # Let human review
```

### Human feedback loop

When a human reviews the design, they may give feedback with refs:

```
"The hero section $${home@.hero} needs more padding"
```

1. `dekit resolve '$${home@.hero}'` → find the file and line range
2. Edit the CSS at that location
3. `dekit screenshot '$${home@.hero}'` → verify the change
